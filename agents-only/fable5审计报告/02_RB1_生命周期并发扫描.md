## 1. 运行时生命周期地图

**存在三条并行的、语义不一致的运行时入口**(关键架构事实):

| 入口 | 拥有者 | inbound 启动 | inbound 停止 | 当前是否活路径 |
|---|---|---|---|---|
| **A. Supervisor**(`sb-core/runtime/supervisor.rs`) | `Arc<RwLock<State>>`(engine+bridge+context) | `bridge.inbounds` → `spawn_blocking(ib.serve())` | `ib.request_shutdown()`(trait 默认 no-op) | **是**(`run.rs` bin + `cli/run.rs` 都走 `run_supervisor`) |
| **B. bootstrap `Runtime`**(`bootstrap.rs::start_from_config` + `runtime_shell.rs`) | `Runtime{ inbounds: Vec<InboundHandle> }` | `inbound_starter::start_inbounds_from_ir` → `tokio::spawn` + channel | `InboundHandle::shutdown()`(channel.send + join) | **零生产调用方**(全仓仅 1 处=自身定义) |
| **C. `sb-core::runtime::Runtime`**(`runtime/mod.rs`) | OS `thread::spawn` workers | `thread::spawn(ib.serve())` | `shutdown(self){ let _ = self; }`(显式 no-op) | admin/test 兼容 shim |

实际生产数据面 = **A**。所有权链:`Supervisor.state: Arc<RwLock<State>>` → `State{ engine, bridge: Arc<Bridge>, context: Context, health: Option<JoinHandle> }`;`Context` 持 9 个 `Arc<Manager>` + `Option<Arc<dyn V2RayServer>>` + sidecars。reload 时 build 新 `Context`/`Bridge`,pre-swap 事务块激活,失败 → `shutdown_failed_reload_context`,成功 → 原子 swap 后 `stop_endpoints/stop_services/shutdown_replaced_context(old)`。graceful shutdown → `handle_shutdown` drain + 全 close。

**adapter inbound 的关键事实**:每个 `XxxInboundAdapter::serve()`(register.rs)**在 `spawn_blocking` 内自建一个 multi-thread tokio runtime**,`rt.block_on(serve_xxx())`,函数返回时 `rt` drop → 该 inbound 的所有内部 spawn(心跳 + per-conn)被 abort。

## 2. 发现清单

### [RB1-01] reload 永远全量重建所有 inbound,与"hot reload 不断连接"承诺矛盾 | P1 | SEMANTIC GAP / ARCHITECTURE DEBT | CONFIRMED
- **evidence**:`supervisor.rs:12` 文档承诺 *"Hot Reload: Applies new configurations without dropping existing connections"* 与 *"Diffing: only restarting changed inbounds"*。但 `handle_reload`(:572-749)无任何 diff 分支:step0 对**每个**旧 inbound 无条件 `ib.request_shutdown()`(:575-577),随后 build 全新 bridge 并重 bind **所有** inbound(:657-664)。diff 仅在 `reload()`(:518-529)用于**日志**(`runtime_diff_from_env()` gated),从不影响行为。
- **trigger**:任何 SIGHUP / watch / provider update 且 config 指纹有变(哪怕加一条 rule)。app 层指纹相同才 skip(`config_load.rs:117`)。
- **impact**:每次 reload 全部监听端口经历 request_shutdown→grace sleep→rebind 窗口,所有客户端连接被切。Go sing-box 是按 inbound diff 增量重启 → **drop-in 行为偏差**。
- **min-verify**:起 socks+http 两 inbound,改一条无关 outbound,SIGHUP,观察两端口都出现 accept 中断窗口而非仅变化项。

### [RB1-02] `request_shutdown` 信号丢失竞态:serve() 尚未注册 stop_tx 时的 reload | P2 | BUG / RACE | HIGH-CONFIDENCE INFERENCE
- **evidence**:adapter inbound 用 `stop_tx: Mutex<Option<Sender>>`,**只有在 `serve()` 跑起来后**才 `*guard = Some(tx)`(register.rs:3705-3708);`request_shutdown` 做 `guard.take()`,若此刻为 `None` 则信号**静默丢弃**(:3725-3728)。inbound 经 `spawn_blocking`(supervisor.rs:228/659)启动 — 调度可被延迟。reload step0(:575-577)对刚 swap 进来的旧 inbound 立即 request_shutdown;graceful shutdown step(:1007-1009)同理。
- **trigger**:reload 间隔极短(连续两次 SIGHUP)或系统负载高使 `spawn_blocking` 未及调度即被 request_shutdown。
- **impact**:该 inbound 的 serve 永不收到 stop → accept 循环不退出 → 端口不释放 → 下一轮同端口 rebind EADDRINUSE,或旧监听悬挂。
- **min-verify**:在 `serve()` 注册 stop_tx 前插入 sleep,reload 后断言端口未释放。注:1200ms grace(`SB_INBOUND_RELOAD_GRACE_MS`,:591-597)缩小但不消除窗口。

### [RB1-03] 生产路径残留 `eprintln!` 调试输出,每次 reload 必触发 | P2 | BUG(可观测性污染) | CONFIRMED
- **evidence**:`register.rs:3724` `eprintln!("HttpInboundAdapter::request_shutdown called")`;`http.rs:308` `eprintln!("serve_http: stop signal received")`;`tuic.rs:847` / `hysteria2.rs:544` connect 失败 `eprintln!`。全非 test-gated。
- **trigger**:每次 reload step0 对 http inbound 调 request_shutdown → 直写 stderr。tuic/hysteria2 在每次 detour bind 失败的**连接热路径**上打。
- **impact**:绕过 tracing/日志级别,污染 stderr,生产噪声;`request_shutdown` 的那条在每次 reload 都打。
- **min-verify**:`cargo run -- run` 起 http inbound 后 SIGHUP,stderr 直接出现该字符串。

### [RB1-04] reload pre-swap 窗口 + 失败回滚后,全局 `RUNTIME_OUTBOUNDS`/`RUNTIME_INBOUNDS` 指向错误 bridge | P1 | BUG / STALE GLOBAL STATE | CONFIRMED
- **evidence**:`build_bridge`(reload 时为**新** context 调用)在构建中即 `registry::install_runtime_outbounds(...)`(bridge.rs:768)与 `install_runtime_inbound_handle`(:836)**覆盖全局 static**(registry.rs:81-83, 137-144)。这发生在 swap **之前**(supervisor.rs:631 build → :702 swap)。该全局被**实时连接路径**读取:`detour.rs:23 runtime_outbounds()` 用于 detour 拨号(`connect_tcp_stream`,热路径)。回滚 `shutdown_failed_reload_context`(:1487-1500)**从不恢复**这两个全局。
- **trigger**:(a) 任意 reload 的 build→swap 窗口;(b) reload 在 `populate_bridge_managers`/Start 阶段失败回滚后——旧 context 仍服务,但全局 outbound handle 已指向被回滚丢弃的新 bridge。
- **impact**:回滚后用 detour 的连接解析到已废弃 bridge 的 outbound 注册表(实例不一致 / 拨号失败);pre-swap 窗口内 detour 拨号短暂指向尚未提交的新表。这是真正的 reload 原子性漏洞,且**不在最近 01A/01B/01C 工作覆盖范围**(那些只管 context/bridge/v2ray 所有权,没碰 registry 全局)。
- **min-verify**:config 含 detour outbound;构造一个会在 Start 阶段失败的 reload(如重复端口 endpoint);回滚后发起 detour 连接,断言 `runtime_outbounds()` 已非旧 bridge。

### [RB1-05] graceful shutdown:request_shutdown 先于 drain 轮询,in-flight 连接被 abort 而非 drain | P2 | SEMANTIC GAP | HIGH-CONFIDENCE INFERENCE
- **evidence**:`handle_shutdown`(:1001-1093)顺序:先对所有 inbound `request_shutdown`(:1007-1009)→ 再 loop 轮询 `active_connections()==0`(:1019-1048)。但 adapter inbound 的 serve 一旦 break accept loop,`block_on` 返回,**嵌套 `rt` 立即 drop**(register.rs:3710-3720),其上所有 per-conn task(http.rs:349 spawn)被 abort。
- **trigger**:任何 shutdown 且存在 adapter inbound 的活动连接。
- **impact**:drain 循环语义形同虚设——request_shutdown 已触发 runtime 拆除,连接在"等待 drain"期间被强断。`active_connections` 归零是因为被 abort,不是因为优雅完成。注:`handle_shutdown` 实际只 break accept loop(stop_rx 仅停 accept,不阻断已 spawn 的 conn),但 `rt` drop 才是真凶——取决于 serve() 返回是否早于 conn 完成。
- **min-verify**:建长连接(大文件转发)中触发 shutdown,grace 足够长,观察连接是否在 deadline 前被断。

### [RB1-06] serve_http 心跳 task 在 bootstrap 路径泄漏(无 handle 无 cancel) | P3 | RESOURCE LEAK | CONFIRMED(adapter 路径已缓解)
- **evidence**:`http.rs:295-301` `tokio::spawn(async move { loop { hb.tick().await } })` — 永久 loop,无 JoinHandle 保存、无 cancel、无 select stop。
- **trigger**:每次 serve_http 调用。
- **impact**:**adapter 路径**(A,生产):跑在嵌套 `rt` 上,serve 返回时随 rt drop 被回收 → 无泄漏。**inbound_starter/bootstrap 路径**(B,`inbound_starter.rs:369` 在 ambient 共享 runtime 上 `tokio::spawn(serve_http)`):心跳 task 永久泄漏于共享 runtime。因 B 路径无生产调用方,实际影响 P3;但 serve_http 是公共 API,任何走共享 runtime 的调用都泄漏。
- **min-verify**:在共享 runtime 上调 serve_http,stop 后断言 tokio task 数未回落。

### [RB1-07] inbound serve 失败(端口 bind 失败)不计入 reload 失败,旧 inbound 已被请求关闭 | P2 | SEMANTIC GAP | CONFIRMED
- **evidence**:reload 中新 inbound 经 `spawn_blocking(ib.serve())`(:657-664),`serve()` 的 `Err`(含 `TcpListener::bind` 失败,http.rs:275)只在 spawn 闭包里 `tracing::error!`(:660-662),**不进 `activation_result`**。pre-swap 事务块(:625-675)不等待、不感知 inbound bind 结果。step0 已对旧 inbound request_shutdown(:575-577)。
- **trigger**:新 config 的 inbound 端口被外部占用 / 与保留端口冲突。
- **impact**:reload 报告 success(swap 完成),旧 inbound 已关、新 inbound bind 失败静默 → **该端口无人监听**,服务空洞,且 reload 返回成功掩盖了它。这正是 `APP-RELOAD-INBOUND-CONTINUITY-01A` 担忧的具体落地形态之一,但根因在 inbound 启动**完全脱离激活事务**。
- **min-verify**:reload 到一个端口被占用的 inbound config,断言 reload 报 success 但端口无监听。

### [RB1-08] `ServiceManager::close()` no-op + 双驱动隐患 | P2 | ARCHITECTURE DEBT | CONFIRMED(已知种子的精确定位)
- **evidence**:`context.rs:70-73` `ServiceManager::close()` 仅 `tracing::debug!` + `Ok(())`,**不遍历 services 调其 close**。真正停 service 全靠 supervisor 显式 `stop_services()`(:1782-1794)。`shutdown_context_inner`(:1509-1545)依赖 service_manager.close() 实为空转——注释(:1478-1480)已承认。
- **trigger**:任何不经 supervisor `stop_services` 而仅靠 `shutdown_context` 的路径。
- **impact**:若未来有人新增一条只调 `shutdown_context`/`shutdown_context_inner` 而漏掉 `stop_services` 的 teardown 路径,service 会静默泄漏(已 Start 的 service 永不 close)。当前 supervisor 三处(reload 成功 :742-743、reload 失败 :1497、graceful :1076-1077)都补了显式 stop,但这是**手工对齐而非结构保证**。EndpointManager.close 有真实 `shutdown()`(:57-60),ServiceManager 不对称。
- **min-verify**:grep 确认所有 `shutdown_context*` 调用点都伴随 `stop_services`;新增路径易漏。

## 4. 对种子 APP-RELOAD-INBOUND-CONTINUITY-01A 的独立判断:**成立,且被低估**

01A 登记的是 "step0 在 reload 结果未知前就 request_shutdown 旧 inbounds"(DEFER)。我的独立结论:

- **成立**:`handle_reload:575-577` 确在 build 新 bridge / 激活之前就关旧 inbound,且 grace sleep(:591-597)固定 1200ms 后才开始 build。期间旧 inbound 已停 accept,新的还没起 → **确定性的服务空洞窗口**,不只是"结果未知"。
- **被低估的三个维度**:
  1. **[RB1-07]**:即便激活事务"成功"swap,inbound `serve()` 的 bind 失败也**完全在事务之外**,reload 仍报 success → 旧 inbound 已关 + 新 inbound 没起 = 永久空洞被成功报告掩盖。01A 只盯 "提前关",没盯 "新 inbound 启动根本不在成功判据里"。
  2. **[RB1-01]**:空洞不是个别变化项,是**全部** inbound(无 diff),每次 reload 全端口重连。01A 隐含按需,实际是全量。
  3. **[RB1-02]**:request_shutdown 信号丢失竞态会让"提前关"退化成"关不掉",空洞变成端口悬挂/EADDRINUSE。

01A 标 DEFER 合理(非崩溃),但其真实影响面 = RB1-07 + RB1-01 + RB1-02 的合集,比"提前关旧 inbound"严重一档:**reload 的 inbound 连续性既无 diff 增量、又不把新 inbound 启动纳入成功判据、还有信号丢失竞态**。最高优先建议先治 RB1-07(把 inbound serve readiness 纳入激活事务)与 RB1-04(回滚恢复全局 registry),这两个能在不报错的情况下导致静默服务中断/拨号错路由。
