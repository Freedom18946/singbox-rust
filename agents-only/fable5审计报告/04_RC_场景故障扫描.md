Live runtime path confirmed: `bin/run.rs` / `cli/run.rs` → `run_engine::run_supervisor` → `run_engine_runtime::supervisor::run_supervisor` → `Supervisor::start_with_registry` (sb-core). `bootstrap.rs`/`start_from_config` is **legacy** (lib.rs:46 says so; no production caller in cli/main). All findings below trace the live path.

Central architectural fact underpinning many scenarios: **inbound bind success is never observed by the supervisor.** Every inbound is launched via `tokio::task::spawn_blocking(move || { ib.serve() })` with the `JoinHandle` discarded and `ready_tx` always `None` (register.rs `.serve(cfg, rx, None)` everywhere; supervisor.rs:228, 402, 659, 829). `serve()`'s `Result` is only `tracing::error!`-logged inside the closure. There is no watchdog, no restart, no readiness gate. Inbounds use plain `TcpListener::bind` with **no SO_REUSEADDR** (mixed.rs:58, http.rs:275, socks/mod.rs:112, sb-core inbound/*).

Confidence: CONFIRMED.**

## 3. Seed Re-Verification

**APP-RELOAD-INBOUND-CONTINUITY-01A (DEFER) → UPHELD / arguably UNDER-RATED.** Independently confirmed at supervisor.rs:572-597 / 762-787: step-0 unconditionally `request_shutdown`s all old inbounds and sleeps the grace window BEFORE the new config's engine/bridge/activation is attempted; on activation failure the run loop logs `Failed` and continues with **no restart of old inbounds** (run_engine_runtime/supervisor.rs:355-367). The seed's framing "continuity gap ≠ leak" is technically correct (the rollback_guard work genuinely closed the *leak*, verified: 9 tests pass), but for the stated **drop-in-replacement reliability goal this is the single most material gap** — Go sing-box validates and builds the new instance before tearing down the old listeners, so a bad reload there is a no-op, whereas here it is a hard outage. I rate it **P1, blocking the reliability face of drop-in replacement**. It is correctly *separate* from the leak work, but should not stay merely DEFER given GUI users routinely reload. Recommend re-classify to an active P1 follow-up (no fix proposed here per scope).

**APP-SIDECAR-POLICY-02A (DEFER, sidecar ownership/policy unclosed) → UPHELD as currently-consistent-but-incomplete.** The liveness/exit policy IS closed and coherent: sidecars (Clash, V2Ray) are auxiliary, log-only `Continue` (sidecar_runtime.rs:284-304; APP-SIDECAR-LIVENESS-01). What remains open is exactly **ownership/observability**, not liveness: (a) Clash is rebuilt per admin-services lifecycle, NOT per supervisor reload — it is structurally outside `handle_reload`, so a reload that changes clash_api config does not re-bind Clash (asymmetry vs V2Ray's reuse path); (b) failed/never-started sidecars are not subscribed by the runtime bridge, so "API down" has no machine-readable signal (RC-05). I rate the policy seed **correct to keep open at P2** — it is not blocking the core proxy, but it does dent GUI observability. Not over- or under-rated; should remain DEFER/POLICY with RC-05 attached as the concrete observability gap.

**SVC-V2RAY-API-01B (DEFER / POLICY REVIEW) → UPHELD, no behavioral defect found in scope.** The V2Ray API server's reload reuse (7dc853ef, `reusable_v2ray_server` supervisor.rs:1318) and pre-swap rollback discrimination (47e15b0b, `same_v2ray_server`/`shutdown_failed_reload_context`) are correctly implemented and tested (`b_rollback_preserves_inherited_v2ray_listener`, `rollback_closes_fresh_v2ray_but_preserves_inherited` — all pass). The only behavioral edge I found is RC-07 (rapid non-equivalent re-enable → transient EADDRINUSE → silent skip), which is a P2 corner case, not a policy break. The async/non-blocking `close()` (v2ray_api.rs:565) is by design and the reuse path is the intended mitigation. This seed is appropriately a policy-review item, not a live bug; **no escalation, no merge/split needed.**

## 4. Confirmed-Robust Scenarios

- **Pre-swap failed-reload rollback (scenario 4, seed 01B):** genuinely complete and well-tested. The single-`async{}`-transaction design funnels every fallible *manager* stage into one rollback; fresh-vs-inherited V2Ray discrimination by pointer identity is correct; cleanup is best-effort/infallible and preserves the original error. All 9 `rollback_guard` tests pass (verified by execution), including real-listener port-release, repeated-failure no-accumulation, and cleanup-error continuation. The commit `47e15b0b` description matches the code.
- **No-leak across repeated failed reloads (scenario 6, leak axis):** old context is never swapped out on failure and stays the stable survivor; each failure cleans only its own fresh resources. No resource accumulation.
- **Shutdown vs reload serialization (scenario 12):** single mpsc event loop + all-state-via-`state.write()`-inside-handlers gives clean serialization; no concurrent mutation path, no deadlock, no double-cleanup. Clash's `reload_sender` shares the same channel.
- **Invalid/parse-failing config (scenario 17 config axis):** rejected in `load_config_with_import_raw` BEFORE `supervisor.reload`, so the running service is fully untouched (the 01A continuity gap does NOT trigger for malformed config — only for validation-passing-but-activation-failing config).
- **Successful-swap ordering (scenario 3):** old resources released strictly after the atomic swap, reused V2Ray correctly excluded from old-context teardown.
- **Sidecar terminal-exit classification (Clash/V2Ray):** outer monitors classify Panicked/ServeError/Cancelled/UnexpectedCompletion exactly once with `#[non_exhaustive]`-safe mapping (no panic/unreachable). This is the well-built counterpart that makes the *inbound* lack of monitoring (RC-03) stand out.

Net: the **leak/rollback face is solid and verified**; the **continuity + inbound-liveness/observability face (RC-01..04, RC-06) is the cluster that blocks drop-in reliability**, with sidecar/DNS observability (RC-05, RC-07, RC-08) as secondary P2s.
