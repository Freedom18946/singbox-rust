## 4. 对最高目标的加权评估:真正阻塞 drop-in 替换的缺口

**绝对阻塞(P0,GUI 主流程当场失败):**
1. **RA-01 TUN schema 拒绝** — GUI 最主要的 TUN 模式配置无法通过启动校验。这是单点致命:GUI 默认 profile 即 TUN,内核根本起不来。
2. **RA-02 启动确认字符串缺失** — 即使配置能解析,GUI 也收不到 `sing-box started`,判定启动失败。这影响**所有**模式(TUN 与非 TUN),是比 RA-01 更普遍的阻断。
3. **RA-03 WireGuard endpoint 无法路由** — 1.12 主推的 WireGuard endpoint 形态在 run 路径硬失败(孤岛接线:`as_outbound_connector` 写好但无人调用)。

**重要削弱(P1):** RA-04(wireguard legacy outbound 永远 stub)、RA-05(TUN 真实数据面仅 smoltcp 且 GUI 无法选择,默认 Manual 为 no-op skeleton)。这两者叠加意味着 **WireGuard 与 TUN 这两个 GUI 关键能力在当前构建下均无可用闭环**。

**结论(对抗式复核):** 非 TUN、纯 clash_api 管理 + 主流代理协议的链路**确实是真闭环且实测可用**——这与文档的乐观叙述一致。但文档/台账掩盖了三个 P0:GUI 的**默认 TUN 配置无法解析**、**启动握手字符串不匹配**、**WireGuard endpoint 孤岛**。其中 RA-01/RA-02 任一未解,GUI 就无法把这个二进制当 sing-box 用。因此当前状态**尚不构成可直接替换 Go sing-box 1.13.13 的 GUI 兼容二进制**;"维护态接近完成" 的表象主要来自 clash_api/选择/持久化等管理面的高完成度,而数据面入口(TUN/WG)与 GUI 进程契约存在结构性断裂。这些缺口是真实的产品路径断裂,而非已接受偏差(accepted-limitation 仅覆盖 redirect/tproxy/tun2socks-stub/ech-quic,**不覆盖** TUN schema、启动字符串、WG endpoint 接线)。
