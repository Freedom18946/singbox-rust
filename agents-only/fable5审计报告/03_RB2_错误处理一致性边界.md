## 1. 危险模式审计汇总

| 模式 | 全 prod 命中 | 过滤后真实风险 | 过滤逻辑 |
|------|------|------|------|
| `unimplemented!`/`todo!` | 0 (本仓代码) | **0** | 仅 vendor/rustls 第三方有,本仓零 |
| `unreachable!` | 1 (本仓代码) | 0 | `shadowtls.rs:1224`,有 prevalidation 守卫 |
| `panic!(` | 125 | **0 真实风险** | 逐一核验:全部在 `#[cfg(test)]` 块或 mutex-poison/build.rs;唯一可疑的 `failpoint.rs:59` 是 env 显式开启的故障注入工具 |
| `.unwrap()` | 2493 | ~0 协议解析路径 | 抽查协议解析文件,均为 mutex-lock unwrap(poison 才 panic)或 env 配置 |
| `.expect(` | 758 | 0 | 抽查 vless/shadowsocks/ssr,均为前置 `is_some()`/同步设值的不变式守卫 |
| `let _ =` 吞错 | 589 | 见 RB2-05/06 少量 | 绝大多数在 best-effort teardown(flush/stop_tx/错误响应后关闭),idiomatic 正确 |

**关键反直觉结论**:本仓库的生产代码在 panic-安全方面**异常健壮** —— 协议解析(trojan/vless/shadowsocks/socks/ssr)、配置解析路径上**没有**可被恶意对端或坏配置触发的 panic/unwrap。坏配置 → 进程崩溃、恶意对端 → DoS 这两类风险**未发现**。这与下面 RB2-01 的发现形成张力:健壮性是靠人工纪律达成的,而非门禁强制的。

## 2. 发现清单

**[RB2-01] workspace 安全 lint 策略对 15/16 crate 是死配置 | P1 | error-handling/process | CONFIRMED**
- evidence: `Cargo.toml:47-50` 定义 `unwrap_used/expect_used/panic = "deny"`;但 `lints.workspace = true` 只在 `crates/sb-tls/Cargo.toml:76-77` 出现。`app`、`sb-core`、`sb-adapters`、`sb-config`、`sb-types` 等均无 `[lints] workspace=true`(逐一确认)。`sb-adapters/Cargo.toml:231` 甚至有 `[lints.rust]` 段却只配 `unexpected_cfgs`、未加 `workspace=true`。`Makefile:15-17` clippy 目标刻意不带 `-D warnings`。
- trigger: 任何开发者在 sb-core/app 等写 `unwrap()`/`panic!()` —— clippy 不报错、CI 不阻断。
- impact: "deny unwrap/panic" 的安全护栏实际只覆盖 1 个 crate。2493 unwrap + 145 panic 与该策略并存即是证据。当前代码靠人工纪律守住健壮性,无机器强制,易随时间漂移。
- min-verify: `grep -rn 'lints.workspace = true' --include=Cargo.toml`(仅 sb-tls 命中)。

**[RB2-02] ServiceManager::close() 是 no-op,但已被一致性补偿(种子复核:graceful 路径不漏)| P3 | boundary-design | DISPROVED(种子隐含的"泄漏"假设)**
- evidence: `crates/sb-core/src/context.rs:70-73` `ServiceManager::close()` 显式 `Ok(())` 空体(对比 `EndpointManager::close()` 调 `self.shutdown()`)。`supervisor.rs:1477` 注释明确登记其为 no-op。
- 但:graceful 关闭路径 `supervisor.rs:1077` 在 `shutdown_context` **之前**先调 `stop_services(&services)`(用 bridge 收集的服务列表逐个 `svc.close()`);reload-失败回滚 `shutdown_failed_reload_context:1498` 同样显式 `stop_services`。`ServiceManager` 本质是状态注册表,真正生命周期走 bridge 的 `services` Vec。
- impact: no-op **不会**导致服务泄漏 —— 所有 8 处关闭路径都有 `stop_services` 补偿(`grep stop_services` 11 处)。这是设计上的职责分离,非缺陷。种子描述准确(它确实是 no-op),但其隐含的"会漏关服务"推论被证伪。
- min-verify: `sed -n '1070,1080p' supervisor.rs` 看 stop_services 在 shutdown_context 前。

**[RB2-03] 配置→IR 路径静默丢弃畸形 FakeIP 掩码 | P3 | error-handling/config | CONFIRMED**
- evidence: `crates/sb-config/src/validator/v2/dns.rs:446` `dd.fakeip_v4_mask = mask.parse::<u8>().ok();`(及 :454 v6)。`validate_v2` 不对 mask 数值/范围做校验(`dns.rs` 无 mask 校验)。run 路径 `Config::from_value:366` → `to_ir_v1` 确实经过此处。
- trigger: 用户配置 `"inet4_range": "198.18.0.0/abc"` 或 `/300` → mask 静默变 `None`,无 error/warn,后续回落默认掩码。
- impact: 用户预期的 FakeIP 网段被静默丢弃,行为与配置不符,无可观测信号。低危(仅 FakeIP 行为降级)。
- min-verify: 给 fakeip range 非法 mask,跑 `config_from_raw_value`,观察不报错且 mask=None。

**[RB2-04] `experimental` 块解析失败静默置空 | P3 | error-handling/config | CONFIRMED**
- evidence: `crates/sb-config/src/validator/v2/top_level.rs:20` `ir.experimental = serde_json::from_value(exp.clone()).ok();`
- trigger: 结构错误的 `experimental` 块 → 整块静默丢弃为 `None`,不报错。
- impact: clash/v2ray API、cache_file 等 experimental 子配置被静默忽略,功能静默不生效。低危但隐蔽。
- min-verify: 给类型错误的 experimental 子字段,观察 IR.experimental=None 且无 issue。

**[RB2-05] system_proxy.enable() 在非支持平台报告成功但实为 no-op | P3 | pattern-consistency/platform | CONFIRMED**
- evidence: `crates/sb-platform/src/system_proxy.rs:285-298` 非 macos/linux/windows/android 分支只 `warn!(...)` 然后 `self.enabled.store(true)` + `Ok(())`。
- trigger: 在 BSD 等小众平台启用 system proxy。
- impact: 返回 Ok 且内部状态标记 enabled,但实际未设置任何系统代理。有 warn 日志(可观测),且 4 大目标平台均真实现,故 P3。同文件 macos 回调 `:265 let _ = update_macos_proxy(...)` 在接口变化时静默吞错(代理更新失败无日志)—— 次要观测缺口。

**[RB2-06] 平台能力不对称:telemetry 静默 None vs 路由能力 loud Err(对比正确)| P3 | pattern-consistency/platform | CONFIRMED(健壮性正例为主)**
- evidence: `wifi.rs:84-88` 非 macos 返回 `None`(stub);`system_proxy.rs:45` 非 macos/linux 默认接口返回 `None`。**对照**:`process/mod.rs:84` 进程匹配在非支持平台返回 loud `ProcessMatchError::PlatformNotSupported`,且 linux/macos/windows/android 全部真实现。
- impact: 路由相关能力(process matching,会影响 route rule 命中)是 loud-fail —— 正确。可选 telemetry(wifi info)静默 None —— 可接受。不对称是**有意且方向正确**的(影响路由的不静默)。

## 4. 全局状态清单及 reload 安全性

reload 机制 = 构建全新 `Context` 后在 `state.write().await` 下 Arc-swap(`supervisor.rs:504/563/703`)。per-context 状态(managers、bridge、services)随 swap 整体替换 —— **架构上 reload-safe**。进程级全局清单:

| 全局 | 类型 | reload 安全性 |
|------|------|------|
| `adapter/registry.rs` RUNTIME_INBOUNDS/OUTBOUNDS | RwLock | 安全(`install_snapshot`/`install_runtime_*` 可替换,reload 重装)——**但安装时机在 pre-swap、回滚不恢复,见 RB1-04** |
| `outbound/registry.rs` GLOBAL | **OnceCell** | `install_global` 用 `let _ = GLOBAL.set()` 静默忽略二次设置;但仅 bootstrap env 驱动、reload 不重调 → 实际无害(DISPROVED 为 reload 缺陷) |
| `outbound/health.rs` STATUS/STATES | OnceCell | register-once,reload 不重建(健康状态跨 reload 累积,符合预期) |
| `metrics/REGISTRY` + 各 metrics Lazy | OnceCell/Lazy | 有意进程级(Prometheus 注册表必须持久);prewarm 用 `Once` 防重复注册 |
| `log/mod.rs` CONFIG | RwLock(OnceLock 包裹) | 可热替换,reload-safe |
| `adapter/clash.rs` CURRENT_MODE | RwLock | 可替换 |
| `inbound/loopback.rs` DETECTOR | OnceLock | 进程级回环检测,无 reload 需求 |

**结论**:无 reload-unsafe 全局状态(注:RUNTIME_* 的**安装时机**问题由 RB1-04 单独登记)。唯一 OnceCell-only-set 的 `outbound/registry.rs::GLOBAL` 因仅 bootstrap-env 驱动而豁免。

## 5. 已确认健壮的部分

- **协议解析 panic 安全**:trojan/vless/vmess/shadowsocks/socks/ssr 解析路径无可被网络数据触发的 unwrap/panic;`expect` 均为前置守卫的不变式。
- **配置坏输入不崩溃**:`config_from_raw_value`(`lib.rs:520`)先 `validate_v2` 严格校验(error 即 `Err` 返回),坏配置不会 panic。
- **生产代码零 `unimplemented!`/`todo!`**;唯一 `unreachable!` 有 prevalidation 守卫。
- **关闭/reload 生命周期**:8 处关闭路径一致补偿 `ServiceManager::close` no-op;reload 用 Arc-swap + 失败回滚 + inherited/fresh 指针区分,逻辑严谨且文档充分。
- **dataplane 错误处理**:`let _ =` 集中在 best-effort teardown(flush-before-close、错误响应后关闭),符合"连接已在失败/关闭"的语义。
- **进程匹配跨平台**:linux/macos/windows/android 全真实现,不支持平台 loud-fail。

**核心张力提示**:健壮性(零生产 panic/解析崩溃)是真实的,但 RB2-01 表明它由人工纪律维持、缺机器门禁强制。这是 P1 流程风险而非 P0 当前缺陷 —— 现状安全,但护栏失效,需复核是否要让 15 个 crate 补 `lints.workspace = true`(届时会暴露现存 2493 unwrap 需逐一评估,工作量大,属 POLICY 决策)。
