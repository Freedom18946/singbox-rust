# post_fable_package02_schema_parity_tun_first

## Status

PLANNED.

## Source Findings

- CAL-01: GUI-generated TUN fields are rejected by strict schema.
- H-4: the same strict-schema gap may exist outside TUN.

## Objective

Make GUI.for SingBox default TUN configs pass Rust config validation while preserving
strict validation and producing a focused Go 1.12.14 option-diff record.

## Implementation Contract

- Extend config validation and IR support for the Go 1.12.14 TUN fields generated by
  the GUI, including at least `address`, `mtu`, `stack`, `strict_route`,
  `route_address`, `route_exclude_address`, and `endpoint_independent_nat`.
- Keep strict unknown-field rejection; do not solve the issue by broadly allowing
  unknown keys.
- Produce an `agents-only` diff note comparing Go 1.12.14 TUN option fields to Rust
  allowed keys and IR fields.
- Add golden/config tests that use a GUI-style TUN inbound.
- Do not implement TUN packet forwarding in this package.

## Out Of Scope

- Mapping GUI stack names to a real TUN backend.
- Creating OS TUN devices or doing live traffic tests.
- Non-TUN schema fixes unless the option-diff shows a field that blocks the same GUI
  launch path.

## Acceptance Criteria

- A GUI-style TUN config passes `app check` or the equivalent config validation path.
- New fields are represented in IR or explicitly documented as accepted/no-op only
  when Go compatibility requires accepting them.
- The Go-vs-Rust TUN schema diff is stored under `agents-only/`.
- Existing config golden tests continue to pass.

## Tests / Verification

- Run `git diff --check`.
