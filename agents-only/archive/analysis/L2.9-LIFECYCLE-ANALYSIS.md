## 一、Go 参考实现分析

### 1.1 四阶段生命周期模型

**文件**: `go_fork_source/sing-box-1.12.14/adapter/lifecycle.go:10-17`

```go
const (
    StartStateInitialize StartStage = iota  // 0: 分配资源/配置（不连接）
    StartStateStart                         // 1: 打开连接/开始监听
    StartStatePostStart                     // 2: 向其他组件注册
    StartStateStarted                       // 3: 全部就绪，清理/GC
)
```

核心接口:
- `Lifecycle { Start(stage) error; Close() error }` — 通用生命周期
- `LifecycleService { Name() string; Lifecycle }` — 内部服务（CacheFile、Clash API 等）
- `SimpleLifecycle { Start() error; Close() error }` — 旧式接口，通过 `LegacyStart()` 映射

### 1.2 Go 完整启动序列

**文件**: `go_fork_source/sing-box-1.12.14/box.go`

#### Phase 1: 构造 (`Box.New()`, lines 97-398)

按以下顺序**创建**（非启动）:
1. Context/registries
2. PauseManager
3. LogFactory
4. CertificateStore → `internalServices`
5. EndpointManager / InboundManager / OutboundManager / DNSTransportManager / ServiceManager
6. DNSRouter
7. NetworkManager / ConnectionManager
8. Router（解析 rules/rule-sets）
9. DNS transports / DNS Router 初始化
10. Endpoints / Inbounds / Outbounds / Services 创建
11. OutboundManager fallback direct（lazy 创建）
12. DNSTransport fallback local
13. CacheFile → `internalServices`
14. Clash API / V2Ray API → `internalServices`
15. NTP → `internalServices`

#### Phase 2: PreStart (`box.go:438-458`)

| 步骤 | Stage | 组件 |
|------|-------|------|
| 1 | Initialize | internalServices (CacheFile, Clash API, V2Ray API) |
| 2 | Initialize | network, dnsTransport, dnsRouter, connection, router, outbound, inbound, endpoint, service |
| 3 | **Start** | **outbound** (拓扑排序!), dnsTransport (拓扑排序!), dnsRouter, network, connection, router |

**关键**: Outbound 在 Start 阶段**第一个**启动，且使用拓扑排序。Inbound 此时尚未启动。

#### Phase 3: Start (`box.go:461-491`)

| 步骤 | Stage | 组件 |
|------|-------|------|
| 4 | Start | internalServices |
| 5 | **Start** | **inbound, endpoint, service** ← 在 outbound 之后 |
| 6 | PostStart | outbound, network, dnsTransport, dnsRouter, connection, router, inbound, endpoint, service |
| 7 | PostStart | internalServices |
| 8 | Started | network, dnsTransport, dnsRouter, connection, router, outbound, inbound, endpoint, service |
| 9 | Started | internalServices |

### 1.3 Outbound 拓扑排序

**文件**: `go_fork_source/sing-box-1.12.14/adapter/outbound/manager.go:93-158`

算法: **迭代多遍依赖解析**（等价于 Kahn's algorithm）:
```
loop:
  for each unstarted outbound:
    if ALL dependencies already started → start it, mark started
  if all started → break
  if no progress → detect cycle via recursive walk → report error
```

**依赖来源**:
| 类型 | 依赖声明方式 |
|------|-------------|
| **Group outbound** (Selector/URLTest) | `options.Outbounds` (成员 tag 列表) 作为 dependencies |
| **Chain outbound** (all protocol types) | `DialerOptions.Detour` (单个 tag) 作为 dependency |

Endpoint 也参与同一轮拓扑排序：`append(outbounds, common.Map(m.endpoint.Endpoints(), ...)...)`

**DNS TransportManager** 使用相同算法 (`dns/transport_manager.go:77-133`)。

### 1.4 Default Outbound 解析

**文件**: `go_fork_source/sing-box-1.12.14/adapter/outbound/manager.go:51-91`

在 `Start(StartStateStart)` 阶段解析:
1. 若 `route.final` 配置了 tag → 查找 endpoint/outbound → 未找到则**致命错误**
2. 若无显式 tag → 使用第一个注册的 outbound
3. 若完全无 outbound → 自动创建 `direct` outbound 作为 fallback

### 1.5 失败处理

**文件**: `go_fork_source/sing-box-1.12.14/box.go:400-436`

策略极其简单: **任何组件启动失败 → 调用 `Box.Close()` 全量关闭**。
- 启动阶段 fail-fast（首个错误立即中止）
- 关闭阶段 error-accumulate（尽量关闭所有组件）
- 无部分回滚或"仅关闭已启动组件"的追踪

### 1.6 关闭序列

**文件**: `go_fork_source/sing-box-1.12.14/box.go:493-512`

大致按启动逆序:
```
service → endpoint → inbound → outbound → router → connection
→ dnsRouter → dnsTransport → network → internalServices → logger
```

### 2.4 关键缺口

#### GAP-1: OutboundManager 完全未接线

`populate_bridge_managers()` (supervisor.rs:1089-1122) 明确跳过了 outbound 注册:
```rust
// Register outbounds with OutboundManager
// ... Skip for now as there's no add_outbound method.
```
Bridge 持有 `Vec<(String, String, Arc<dyn OutboundConnector>)>`，但 OutboundManager 持有 `HashMap<String, OutboundHandler>`（需要 `OutboundAdapter`），两者类型不匹配。

#### GAP-2: 依赖图从未构建

`add_dependency()` 从未被调用。两遍构建（先具体 outbound，后 selector/urltest）是唯一的排序手段。这对以下场景不安全:
- **Chain outbound**: outbound A 的 `detour` 指向 outbound B → A 必须在 B 之后启动
- **嵌套 group**: selector A 的成员包含 urltest B → A 依赖 B

#### GAP-3: OutboundIR 缺少 `detour` 字段

Go 的 `DialerOptions.Detour` 允许 outbound-to-outbound 链式代理。Rust 的 `OutboundIR` (sb-config) **没有 `detour` 字段**，无法表达 outbound 链依赖。

#### GAP-4: Inbound 启动时序不正确

当前 Supervisor 在步骤 10 直接 `spawn_blocking(ib.serve())`，与 outbound 启动（步骤 7 的 manager Start）之间无因果顺序保证。Go 明确在 `preStart()` 中先启动 outbound，然后在 `start()` 中再启动 inbound。

#### GAP-5: 无启动失败回滚

当前 `start_endpoints()` 和 `start_services()` 在单个组件失败时仅 `break`（跳过该组件后续 stage）但继续启动其他组件。Go 的策略是 fail-fast + 全量 `Close()`。

#### GAP-6: OutboundConnector 无生命周期

`OutboundConnector` trait 不含 `start(stage)` / `close()` 方法。Bridge 中的 outbound 是裸的 `Arc<dyn OutboundConnector>`，没有生命周期管理能力。需要要么:
- (a) 扩展 `OutboundConnector` 添加 lifecycle（破坏性大）
- (b) 使用 `OutboundAdapter` wrapper（已定义但未使用）
- (c) 在 Bridge 中存储 `Arc<dyn OutboundAdapter>` 而非 `Arc<dyn OutboundConnector>`

1. **最大风险: 双路径问题** — Supervisor 路径和 legacy bootstrap 路径并存。L2.9 改动需要在 Supervisor 路径实施，但必须确保 bootstrap 路径不回归。理想情况下应废弃一个路径。

3. **L2.9.6 detour 可延后** — outbound chaining 在 GUI.for 使用场景中不常见（用户通常不手写 chain proxy），且 IR 层改动范围大。可在 L2.9 核心完成后再补。

## 五、关键设计决策点

### Q1: OutboundManager 注册方式

**选项 A**: Bridge 构建时同步注册到 OutboundManager
- 优点: 简单，一处改动
- 缺点: Bridge 持有 `Arc<dyn OutboundConnector>`，需要 wrapper 转为 `OutboundAdapter`

**选项 B**: 新增 `OutboundManager::add_connector_with_deps(tag, connector, deps)`，直接存 legacy_connectors + 依赖
- 优点: 不需要 OutboundAdapter wrapper
- 缺点: 绕过 lifecycle（legacy_connectors 无 start/close）

**选项 C**: 在 Bridge 内部实现拓扑排序，不走 OutboundManager
- 优点: 最小改动，不动 OutboundManager
- 缺点: 拓扑排序代码重复，与 Go 架构偏差更大

### Q2: Inbound 启动时序保证

**选项 A**: Supervisor 中显式分两步 — 先 `start_outbounds()`（拓扑排序），再 `start_inbounds()`
- 优点: 与 Go 对齐，清晰
- 缺点: 需重构 Supervisor::start()

**选项 B**: 使用 barrier/dependency 保证顺序但不改代码结构
- 优点: 最小代码改动
- 缺点: 复杂，不如 A 直观

### Q3: 启动失败策略

**选项 A (Go 模式)**: 任何组件失败 → 全量 Close() → 返回错误
- 优点: 简单、安全、与 Go 一致
- 缺点: 一个非关键 outbound 失败会导致整体启动失败

**选项 B**: 非关键组件失败 warn + 继续，关键组件失败才 abort
- 优点: 更健壮
- 缺点: 需定义"关键"/"非关键"
