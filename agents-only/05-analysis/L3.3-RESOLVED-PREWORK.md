# L3.3 Resolved 完整化（PX-015）前置信息收集与差距分析

*更新时间*: 2026-02-09  
*范围*: **信息收集/差距分析**（不做代码实现）

---

## 1. 目标与结论摘要

L3.3 的核心目标是补齐 D-Bus 接口 `org.freedesktop.resolve1.Manager` 的 **Resolve\*** 方法族，并让这些 Resolve\* 调用走 **DNSRouter**（带入站元信息），从而对齐 Go 的 resolved 行为：

- `ResolveHostname`
- `ResolveAddress`
- `ResolveRecord`
- `ResolveService`

同时，当前 Rust 的 `resolved` 服务实现与 Go 的运行模型存在结构性偏差（“既要替代 resolved 又要连接真实 resolved”），这会直接阻塞 L3.3 的落地，需要在实现阶段一并纠正（见第 3 节）。

## 2. 参考实现与对照文件

### Go（权威行为来源）

- `go_fork_source/sing-box-1.12.14/service/resolved/service.go`
  - resolved service 在 **system bus** 上导出 `org.freedesktop.resolve1.Manager` 并请求 name `org.freedesktop.resolve1`。
  - 若真实 systemd-resolved 正在运行（name 已存在），Go 直接报错退出：`"maybe real resolved is running"`。
  - DNS stub listener 的 TCP/UDP 请求直接调用 `dnsRouter.Exchange(...)`（不是 system resolver）。
- `go_fork_source/sing-box-1.12.14/service/resolved/resolve1.go`
  - `ResolveHostname/ResolveAddress/ResolveRecord/ResolveService` 的 D-Bus 方法实现。
  - 通过 `GetConnectionUnixProcessID` + `/proc/<pid>` 采集 sender 进程信息，写入 metadata 并用于日志前缀。
- `go_fork_source/sing-box-1.12.14/service/resolved/transport.go`
  - transport 消费 per-link 配置，按域名匹配选择 link，上游 dialer 绑定 `BindInterface=link.iif.Name`，并对 A/AAAA 做并行查询。

### Rust（当前落点）

- D-Bus server：`crates/sb-adapters/src/service/resolve1.rs`
  - 已实现：`SetLinkDNS/SetLinkDNSEx/SetLinkDomains/SetLinkDefaultRoute/SetLinkDNSOverTLS` + 少量 stub。
  - **缺失**：所有 Resolve\* 方法。
  - 已使用 zbus，且可通过 `#[zbus(header)] MessageHeader` 获取 sender 信息（zbus 3.15 支持）。
- Resolved service：`crates/sb-adapters/src/service/resolved_impl.rs`
  - 当前实现包含“连接真实 systemd-resolved 并调用 ResolveHostname”的 D-Bus client 逻辑，以及 UDP stub listener。
  - 目前 stub listener 走 `DnsResolver::resolve()` / `ResolveHostname` client fallback，不走 DNSRouter。
- 共享状态与 transport：`crates/sb-core/src/dns/transport/resolved.rs`
  - `RESOLVED_STATE: Arc<Resolve1ManagerState>` 保存 per-link 配置与 default route 顺序。
  - `Resolve1ManagerState` 里已有 `dns_router: RwLock<Option<Arc<dyn DnsRouter>>>`，但目前语义写的是“cache management”。
- DNSRouter 接口：`crates/sb-core/src/dns/dns_router.rs`
  - `exchange(&DnsQueryContext, message: &[u8]) -> Result<Vec<u8>>`
  - `lookup(&DnsQueryContext, domain: &str) -> Result<Vec<IpAddr>>`
  - `DnsQueryContext` 当前可携带 `inbound/client/transport/fakeip`（无 process info 字段）。

## 3. 关键差距与阻塞点（Rust vs Go）

### 3.1 Resolve\* 缺失（L3.3 直接目标）

Rust 的 `crates/sb-adapters/src/service/resolve1.rs` 当前没有实现：

- `ResolveHostname(ifIndex, hostname, family, flags) -> (addresses, canonical, outflags)`
- `ResolveAddress(ifIndex, family, address, flags) -> (names, outflags)`
- `ResolveRecord(ifIndex, hostname, qClass, qType, flags) -> (records, outflags)`
- `ResolveService(ifIndex, hostname, sType, domain, family, flags) -> (srv, txt, canonicalName, canonicalType, canonicalDomain, outflags)`

Go 的 Resolve\* 都是 **dnsRouter** 路径，并附带 inbound metadata（resolved 的 tag/type + sender process info）。

### 3.2 ResolvedService 运行模型与 Go 相悖（实现阶段必须纠正）

Go 的 resolved service 是“替代实现”：在 system bus 请求 `org.freedesktop.resolve1` name，若真实 resolved 运行则直接报错退出。

Rust 当前 `crates/sb-adapters/src/service/resolved_impl.rs` 同时做了两件互斥的事：

- `connect_dbus()` 验证/连接真实 `org.freedesktop.resolve1`（意味着真实 systemd-resolved 存在）。
- `start()` 又启动 D-Bus server 并 `request_name("org.freedesktop.resolve1")`（真实 resolved 存在时必然冲突）。

并且还有一个明显的生命周期问题：

- `StartStage::Start` 里检查 `self.connection` 必须已初始化，但 `Initialize` 阶段并未调用 `connect_dbus()` 进行初始化，导致按字面逻辑会直接报错 `"D-Bus connection not initialized"`。

这意味着：即使只做 Resolve\*，也需要先把 resolved service 的“谁是 D-Bus owner / 是否连接真实 resolved / DNS 请求走哪条链路”定案为 **Go 同款替代模型**，否则实现无从验证。

## 4. Go 的 Resolve\* 语义要点（实现对齐点）

### 4.1 sender metadata（日志与策略一致性）

Go 在每个 Resolve\* 内通过 sender：

- `org.freedesktop.DBus.GetConnectionUnixProcessID(sender)` 获取 pid
- 读取 `/proc/<pid>/exe`、`/proc/<pid>/status` 提取 path/uid/user
- 写入 inbound metadata（resolved tag/type + process info）
- 日志带前缀（basename(processPath) / user / uid）

Rust 侧如果要对齐：

- zbus 3.15 支持在 interface 方法参数中加入 `#[zbus(header)] header: zbus::MessageHeader<'_>`，从 header 取 sender unique name。
- 通过 `zbus::fdo::DBusProxy` 调用 `GetConnectionUnixProcessID`（或直接构造 proxy call）获取 pid。
- `/proc` 解析只在 Linux 下启用，失败则降级为仅 inbound tag/type。

### 4.2 ResolveHostname

Go 行为：

- `family` 映射 DomainStrategy（AF_UNSPEC/AF_INET/AF_INET6）
- `dnsRouter.Lookup(ctx, hostname, options{LookupStrategy})`
- 返回 `Address{IfIndex, Family, AddressBytes}` 数组 + canonical name

Rust 落地建议：

- 使用 `DnsQueryContext::with_inbound(<resolved_tag>)` 构造 ctx。
- 若没有“策略参数”，则：
  - `AF_INET`/`AF_INET6`：`lookup()` 后按 v4/v6 过滤。
  - `AF_UNSPEC`：直接使用 `lookup()` 全量结果。

### 4.3 ResolveAddress（PTR）

Go 行为：

- 将输入二进制地址构造 PTR 域名（`in-addr.arpa` / `ip6.arpa`）
- `dnsRouter.Exchange(...)` 发送 PTR query
- 解析 Answer 中的 PTR 记录，返回 Name 列表

Rust 落地建议：

- 构造 wire-format PTR query（见第 5 节的 query builder 复用点）。
- 使用一个“最小 DNS 响应解析器”读取 Answer 区 RRs，并筛选 `TYPE=PTR(12)`，提取 target name。

### 4.4 ResolveRecord（任意 qclass/qtype）

Go 行为：

- 构造 query：`Question{Name: FQDN(hostname), Qtype, Qclass}`
- `dnsRouter.Exchange(...)`
- 遍历 Answer，`PackRR(rr, data, 0, nil, false)` 生成**独立 RR** bytes，写入 `ResourceRecord.Data`

Rust 落地建议：

- 不能直接把 response 里某段切片当作 RR data：其中可能包含压缩指针（指向 message 内偏移），脱离原 message 会失效。
- 需要“解压缩并重打包”：
  - 用 name 解压缩（读取 label/pointer）得到完整 name 字符串。
  - 把 `TYPE/CLASS/TTL/RDLENGTH/RDATA` 重新编码成一个独立 RR bytes（name 用全标签编码，不使用 pointer）。

### 4.5 ResolveService（SRV + TXT）

Go 行为：

- 拼出 `serviceName = <hostname>.<sType>.<domain>.`（补齐点号）
- `Exchange(SRV)` + `Exchange(TXT)`
- 对每条 SRV answer：
  - 解析 priority/weight/port/target
  - `dnsRouter.Lookup(target, family strategy)` 填充 Addresses
  - 若 Answer 中存在 `CNAME` 且 `Name == target`，写入 `CNAME`
- TXT：对每条 TXT RR 使用 PackRR 得到 bytes，作为 TXTRecord（`[]byte`）返回
- canonicalName/type/domain：分别 canonicalize

Rust 落地建议：

- SRV RDATA 解析需要最小实现（priority/weight/port + target name 解压缩）。
- TXT record 可直接“解压缩并重打包整条 RR”为 bytes（RDATA 本身无需理解）。

## 6. 建议的最小实现切片（L3.3 的“闭环”定义）

### 6.1 接线闭环（必要前置）

- resolved service 必须像 Go 一样成为 `org.freedesktop.resolve1` 的 owner（真实 resolved 运行时明确报错）。
- resolved 的 DNS 请求路径（TCP/UDP stub listener）应走 `ServiceContext.dns_router`（而非 `DnsResolver` / 调用真实 resolved）。
- `Resolve1ManagerState` 需要能拿到 dns_router（用于 Resolve\* 与 FlushCaches），并且 resolved service start 时完成 wiring。

### 6.2 Resolve\* 方法闭环

在 `crates/sb-adapters/src/service/resolve1.rs` 增加 4 个 Resolve\* 方法，实现：

- 基于 DNSRouter 的 `lookup/exchange`
- 处理 family/qtype/qclass 的基本语义
- 最小可用的 response 解析与 RR 重打包
- sender metadata：至少 inbound tag/type（进阶再补 pid/user/processPath）

## 7. 测试建议（不依赖真实 D-Bus）

- 单元测试（纯函数）：
  - PTR domain 构造正确性（IPv4/IPv6）
  - DNS name 解压缩与 RR 重打包（含压缩指针 case）
  - SRV RDATA 解析
- 组件测试（feature-gated）：
  - 注入 fake `DnsRouter`，验证 Resolve\* 返回的 tuple 结构与过滤逻辑
- 集成测试（可选，Linux-only）：
  - 在没有真实 resolved 的环境启动 D-Bus server 并用 zbus client 调用 Resolve\*

## 8. 后续需要确认的开放问题（实现前）

- system bus 上请求 `org.freedesktop.resolve1` name 是否需要额外权限/PolicyKit（Go 代码假设可行；Rust 需复现实测）。
- Rust 的 `DnsQueryContext` 是否需要扩展 process info 字段，还是只用于日志（路由层是否消费该信息）。
- `ResolveService` 的返回结构在 D-Bus 侧的 exact signature（Go 用 godbus struct 映射；Rust 用 zbus tuple/struct 需对齐）。
