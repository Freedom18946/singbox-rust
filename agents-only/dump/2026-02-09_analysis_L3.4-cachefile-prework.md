## 2. Rust 现状（按“是否真正生效”分层）

### 2.1 IR（配置层）

文件：`crates/sb-config/src/ir/experimental.rs`

当前 `CacheFileIR` 字段：

- `enabled: bool`
- `path: Option<String>`
- `store_fakeip: bool`
- `store_rdrc: bool`
- `rdrc_timeout: Option<String>`

缺失：`cache_id`（L3.4 明确要补）。

### 2.2 Runtime 接线

文件：`crates/sb-core/src/runtime/supervisor.rs`，函数 `wire_experimental_sidecars()`

- 仅当 `experimental.cache_file.enabled == true` 时创建 `CacheFileService::new(cache_cfg)` 并 `context.with_cache_file(cache_svc)`。
- `Context.cache_file` 类型是 `Option<Arc<dyn CacheFile>>`（trait 仅包含 clash mode/selected/expand 相关方法）。

### 2.3 CacheFileService（sled 后端）

文件：`crates/sb-core/src/services/cache_file.rs`

树（sled tree）命名现状：

- `meta`：`clash_mode` key（字符串）
- `selected`：key=group, val=selected
- `expand`：key=group, val=[0|1]
- `rulesets`：key=tag, val=bytes（当前仅测试覆盖，未发现调用方）
- `fakeip_domain` / `fakeip_ip` / `fakeip_meta`：FakeIP 映射与 counters（当前仅测试覆盖，未发现调用方/未接线到 allocator）
- `rdrc`：既存储 “entry(ips+expires_at)” 又存储 “rejection(transport\\0domain\\0qtype -> expires_at)”（当前仅测试覆盖；dns/config_builder 注释明确写的是 “future RDRC”）

### 2.4 “真正生效”的调用路径（现状）

1. Clash/GUI 可见的持久化（生效）
   - `CacheFile` trait 方法：`get/set_clash_mode`, `get/set_selected`, `get/set_expand`
   - 这些由 SelectorGroup / Clash handler 路径在运行时调用（已在 L2.6/L2.7 对齐中接入）。

2. FakeIP/RDRC/ruleset（当前不生效）

   - `check_rdrc_rejection()` / `save_rdrc_rejection()`：无调用方（同样仅 tests）。
   - `store_rule_set()` / `get_rule_set()`：无调用方（仅 tests）。
   - FakeIP allocator `crates/sb-core/src/dns/fakeip.rs` 有 `set_storage()` API，但仓库内没有任何地方调用它（意味着即使启用 cachefile，也不会用于 FakeIP 复用）。

## 3. Go 行为要点（用于定义 L3.4 “对齐到什么程度”）

### 3.1 配置语义（文档）

文件：`.../docs/configuration/experimental/cache-file.md`

- `path` 为空时，默认使用 `cache.db`
- `cache_id`：不为空时，“configuration specified data will use a separate store keyed by it”

### 3.2 cache_id 的存储结构（实现）

文件：`experimental/cachefile/cache.go`

- `selected/group_expand/rule_set/rdrc2` 等 bucket 通过 `c.bucket(tx, key)` 访问：
  - `cache_id == nil`：使用顶层 bucket
  - `cache_id != nil`：使用顶层 bucket `t.Bucket(c.cacheID)` 下的子 bucket（由 `createBucket()` 创建）
- `clash_mode` 是特殊的：
  - 使用顶层 bucket `clash_mode`
  - value 的 key 使用 `cacheIDDefault` 或 `c.cacheID`（即 mode 是按 key 隔离，而不是按子 bucket 隔离）
- FakeIP bucket（`fakeip_*`）在 `fakeip.go` 中**不经过** `c.bucket()`：
  - 目前实现上看，FakeIP 没有按 `cache_id` 隔离（这一点与文档表述可能存在张力，需要在 Rust 侧定一个明确策略）。

### 3.3 FakeIP “metadata debounce” 的真实含义

文件：`experimental/cachefile/fakeip.go`

- metadata 的写盘是 debounced：
  - `FakeIPSaveMetadataAsync()` 使用 `time.AfterFunc(interval)`，反复调用会 `Reset(interval)`，最终只写一次 `FakeIPSaveMetadata()`
- FakeIP 映射写盘是 async（但不是合并写）：
  - `FakeIPStoreAsync()` 维护 in-memory overlay（saveDomain/saveAddress4/saveAddress6），并起 goroutine 写入 BoltDB

### 3.4 RDRC 存储模型

文件：`experimental/cachefile/rdrc.go`

- bucket=`rdrc2`，且按 transportName 再建子 bucket
- key = `qType(u16 BE) + qName(bytes)`
- value = `expiresAt(u64 BE)`
- `SaveRDRCAsync()` 同样维护 in-memory map，避免重复写/读到不一致

### 3.5 ruleset 缓存模型

文件：`experimental/cachefile/cache.go`

- ruleset 存取使用 `adapter.SavedBinary`（不是纯 bytes）
- Rust 当前 `remote.rs` 使用文件缓存 `md5(url).{srs,json}` + `.meta.json(ETag/Last-Modified)`，逻辑上已具备“条件请求 + 失败回退缓存”能力。

## 4. 差距清单（L3.4 需要处理的点）

### 4.1 P0：cache_id（IR + 存储隔离）

- Rust IR 没有 `cache_id`
- Rust sled tree/key 没有 namespace 概念
- 需要可验证的隔离语义：
  - 同一个 DB path 下，cache_id=A 的 selected/mode/expand 不影响 cache_id=B
  - `cache_id == None/""` 必须保持旧数据可读（兼容旧 cache.db）

### 4.2 P1：path 默认值

Go 文档：`path` 为空默认 `cache.db`

Rust 当前：
- `enabled=true` 且 `path=None` 时会退化为 Memory backend（行为差异）
- `path=Some("")` 的语义未定义（大概率会尝试打开空路径）

这属于“容易踩坑”的配置差异，建议纳入 L3.4（实现成本低，收益高）。

### 4.3 P1：FakeIP metadata debounce（以及“要不要接线”）

现状问题是两层的：
- “debounce”能力缺失（CacheFileService 每次 store 都直接 `insert`）
- 更关键的是：FakeIP 持久化根本没被接线到 allocator（`fakeip::set_storage()` 没有调用方）

因此 L3.4 若要兑现 “FakeIP metadata debounce”，需要先确认目标是：
- 仅实现 CacheFileService 内部的 debounced 写策略（先把能力做完，等后续接线）
- 还是同时补齐 FakeIP persistence 接线与（可能的）metadata/counters 模型

### 4.4 P1：ruleset cache 接线点定案

当前 Rust：
- runtime ruleset caching 走 `RuleSetManager(cache_dir)` + `remote.rs` 文件缓存
- `CacheFileService.store_rule_set/get_rule_set` 未被使用

L3.4 需要至少做到：
- 选项 A：明确 “ruleset 缓存权威 = file cache”，CacheFileService 不接线（并在文档/实现注释中固定下来）
- 选项 B：将 ruleset 下载链路接到 CacheFileService（需要定义 data model，至少要覆盖 ETag/Last-Modified/format/source）

## 5. 设计决策点（实现前需要你确认的 3 个点）

1. `cache_id` 的覆盖范围
   - 仅 Clash 相关（mode/selected/expand）隔离（与当前 L3 文档的验收文字一致）
   - 还是连同 RDRC/ruleset/FakeIP 一起隔离（更符合 Go 文档措辞，但 Go 代码对 FakeIP 未隔离）

2. FakeIP：本阶段是否要“真正接线”
   - 只做 CacheFileService 的写盘策略（debounce），暂不接线 allocator
   - 同时接线 `fakeip::set_storage(cache_svc)`，让映射在重启后可复用（并决定是否补齐 counters/metadata 的持久化语义）

3. ruleset caching：A 还是 B
   - A：保持当前 file cache 为权威，CacheFileService 不接线
   - B：对齐 Go 的 bucketRuleSet（意味着要把 SavedBinary 或等价结构塞进 CacheFileService，并修改 RuleSetManager/remote.rs 的调用链）

## 6. 建议的最小落地路径（不含具体实现，只列检查点）

### Phase 1（建议先做，收益最大，风险最小）

- IR：`CacheFileIR.cache_id: Option<String>`
  - 解析空字符串为 None（与 Go “empty means default store” 对齐）
- CacheFileService：实现 namespace
  - 推荐 tree 维度隔离：`meta/{cache_id}`, `selected/{cache_id}`, `expand/{cache_id}`, `rulesets/{cache_id}`, `rdrc/{cache_id}`, etc.
  - 兼容策略：`cache_id==None` 时继续使用旧 tree 名称（无前缀），保证读旧数据。
- path 默认值：`enabled=true && (path is None or empty)` 时使用 `cache.db`

验收（最小可测）：
- 同一路径下，cache_id A/B 的 `clash_mode/selected/expand` 不互相覆盖（单测：TempDir + 两个 svc）。

### Phase 2（按你在第 5 节的选择推进）

- FakeIP metadata debounce：定义“哪些写需要 debounce”（至少 counters/metadata，是否包含映射）
- ruleset cache：定案 A/B，并补齐调用链（若 B，需要定义结构与迁移策略）

## 7. 建议补充的“本地可验证”测试清单

- `cache_id` 隔离测试（sled persistence）：
  - 同一 DB 路径，svc(A) set_selected("g","x")，svc(B) get_selected("g") == None
  - svc(A) set_clash_mode("Rule")，svc(B) get_clash_mode() == None
- path 默认值测试：
  - enabled=true + path=None 时应该打开 `cache.db`（或至少不降级为纯内存），并能重启后读回
- debounce 测试（如果 Phase 2 做）：
  - 用 tokio time pause/advance 控制时间，断言写入次数被合并（需要在实现中暴露可观测计数或可注入 writer）
