# 01 总体与配置入口

有些内核，像一座城：协议是街道，路由是交通灯，DNS 是门牌号，日志是夜里的灯火。sing-box 的“内核”要做的，就是让这座城在不同系统、不同网络与不同协议之间，仍保持秩序与可预测性。

## 1) 产品定位与边界

- MUST：内核应定位为“通用代理平台（universal proxy platform）”，提供多协议入站/出站、路由与 DNS 协同能力。  
- SHOULD：提供清晰的模块边界：配置解析层、协议栈层、路由决策层、DNS 子系统、透明代理/系统集成层、服务/实验扩展层。

## 2) 配置文件结构（JSON）

### 2.1 顶层结构

- MUST：支持 JSON 配置文件。
- MUST：支持以下顶层键位（空对象/空数组允许）：  
  `log`、`dns`、`ntp`、`certificate`、`endpoints`、`inbounds`、`outbounds`、`route`、`services`、`experimental`。  
- SHOULD：配置解析应提供**明确的错误定位**（路径、数组下标、字段名），便于运维排障。

### 2.2 配置工具链（CLI）

- MUST：提供配置校验命令：`sing-box check`（静态语义检查）。  
- MUST：提供配置格式化命令：`sing-box format -w -c config.json -D config_directory`。  
- MUST：提供配置合并命令：`sing-box merge output.json -c config.json -D config_directory`。  

### 2.3 Rule-Set 工具链（CLI）

- MUST：支持将“规则源格式（source format）”编译为二进制规则集：  
  `sing-box rule-set compile [--output <file-name>.srs] <file-name>.json`
- SHOULD：支持把外部规则格式（当前官方仅明确：AdGuard DNS Filter）转换为二进制规则集：  
  `sing-box rule-set convert --type adguard [--output <file-name>.srs] <file-name>.txt`

## 3) 运行时装配与依赖关系（高层）

- MUST：启动时应按依赖顺序装配：  
  证书信任库/证书加载 → 日志 → DNS → 路由/规则集加载与订阅 → Inbounds/Outbounds/Endpoints → Services/Experimental。
- SHOULD：支持“热更新/自动重载”能力：当规则集（本地文件）或证书文件发生改变时可自动重载（官方文档对部分字段有明确说明）。
- SHOULD：提供版本迁移提示：当检测到已弃用字段/模块，输出迁移建议（见 Migration）。

## 4) 典型验收点

- 能用最小结构空配置启动并给出清晰报错（例如缺少必填 inbounds/outbounds）。
- `check/format/merge` 可在 CI 内执行，且输出稳定。
- rule-set compile/convert 可用于构建发布流水线。
