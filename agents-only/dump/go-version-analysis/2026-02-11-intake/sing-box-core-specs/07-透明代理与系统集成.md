# 07 透明代理与系统集成（TUN / Redirect / TProxy / 平台差异）

透明代理的本质，是把“代理”从应用层的选择，变成系统层的事实：应用以为它直接连向世界，实际上世界被你重新布线。

## 1) 目标

- MUST：在支持的平台上提供三类透明接入：
  1) TUN 虚拟网卡接管（`tun` inbound）  
  2) NAT/REDIRECT 方式重定向（`redirect` inbound）  
  3) Linux TProxy 方式接管（`tproxy` inbound）
- MUST：透明模式下能够**恢复原始目的地址**（否则路由与策略失去语义）。

## 2) 与路由系统的关键联动

### 2.1 ICMP（ping）可路由化

- MUST：支持 `icmp` 作为 Route Rule 的 `network` 匹配值。  
- MUST：当 ICMP 流量来自 `TUN` / `WireGuard` / `Tailscale` 入站时，可路由到 `Direct` / `WireGuard` / `Tailscale` 出站（文档明确该通路）。

### 2.2 域名语义的获取：FakeIP + reverse mapping

- SHOULD：透明代理中，为提升“域名可见性”，建议支持 FakeIP 与 reverse mapping：  
  - 应用发起 DNS → 记录域名与返回 IP 映射  
  - 后续连接命中该 IP 时，路由系统可还原域名用于规则匹配
- MUST：在 macOS 等“系统代理并缓存 DNS”的环境，明确 reverse mapping 的限制与风险（文档指出可能会出现问题）。

## 3) 平台网络状态感知

- SHOULD：支持 Wi‑Fi State，允许路由规则按 `wifi_ssid` / `wifi_bssid` 匹配（平台支持范围见官方说明）。
- SHOULD：支持 `network_is_expensive` / `network_is_constrained` 等网络状态字段，作为移动端/节能场景的策略输入。

## 4) DNS 平台接口差异（与透明代理强相关）

- SHOULD：对 `dns.server.type=local` 的实现，需要遵循官方对 Android/macOS 等平台的行为描述：  
  - Android 图形客户端可能只能通过平台接口解析并存在版本限制  
  - macOS Network Extension 场景下可能优先 DHCP 等

## 5) 验收清单

- 透明代理三种入站能正确获取 original destination 并完成路由。
- ping/ICMP 能在 TUN 场景下按规则路由，且不会误伤 TCP/UDP。
- 在支持的平台上，wifi_ssid/bssid 规则生效；不支持的平台上给出明确告警或自动降级。
