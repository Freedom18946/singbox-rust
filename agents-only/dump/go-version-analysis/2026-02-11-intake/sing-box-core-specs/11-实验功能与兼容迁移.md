# 11 实验功能与兼容迁移（Experimental / Migration）

系统越演化，越需要两种力量并存：一是向前的创造（experimental），二是向后的慈悲（migration）。否则用户会在某次升级里，突然失去路径。

## 1) Experimental 模块目标

- MUST：支持 `experimental` 配置段作为可选能力集。
- SHOULD：experimental 能力应：
  - 默认关闭
  - 开启后对行为影响可解释
  - 在日志中显式标注（便于排障）

## 2) Cache File（实验：运行态缓存）

- SHOULD：支持 `experimental.cache_file`：将运行过程中产生的数据落盘以提升体验与稳定性。  
  文档中明确出现的缓存点包括（示例）：  
  - FakeIP 相关缓存  
  - RDRC（与 DNS 规则/地址过滤缓存相关的能力提示）
- MUST：缓存文件应具备：
  - 原子写入/崩溃恢复（避免损坏导致启动失败）
  - 版本化（升级可迁移或丢弃并重建）
  - 可关闭与可清理机制

## 3) Clash API（实验：控制面兼容）

- SHOULD：支持 Clash API，用于：
  - 查询出站组（selector/urltest）状态
  - 切换 selector 的当前选择
  - 提供更丰富的运行态信息给上层客户端/面板
- MUST：Clash API 服务应具备最小安全默认值（localhost + 鉴权/密钥等，具体以文档字段为准）。

## 4) V2Ray API（实验：另一控制面）

- COULD：支持 V2Ray API（以官方字段为准），用于兼容 V2Ray 生态的控制/统计接口。
- SHOULD：与日志与权限控制联动，避免无意暴露运行态信息。

## 5) Migration（弃用与迁移策略）

- MUST：对被标注 deprecated 的配置项，在 `check` 与运行时给出明确提示，并指向替代方案。
- SHOULD：提供迁移指南/自动迁移工具（至少在文档层/提示层），覆盖典型迁移路径：
  - 旧 outbound DNS 规则 → domain resolver / resolve action
  - 旧 special outbounds（block/dns 等）→ rule actions
  - 旧 inbound 字段 → rule actions / route options
  - direct outbound 的 destination override → route options
  - WireGuard outbound → endpoint

> 备注：文档明确给出 legacy DNS servers 的弃用与移除版本，以及多项迁移示例；实现需以该路线为准。

## 6) 验收清单

- experimental 各模块可独立启用/禁用，互不污染。
- cache_file 开启后重启仍能复用缓存；关闭后不产生文件。
- clash_api/v2ray_api 在安全默认下可用，且不会被误暴露到公网。
- 对 deprecated 配置给出强提示与可操作迁移建议。
