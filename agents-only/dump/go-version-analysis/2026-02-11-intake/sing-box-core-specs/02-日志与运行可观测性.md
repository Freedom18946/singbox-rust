# 02 日志与运行可观测性

当系统复杂到足以“欺骗”人的直觉时，日志就不再是附属品——它是你和真相之间唯一可靠的绳索。

## 1) 日志模块目标

- MUST：在配置错误、网络错误、协议握手失败、DNS/路由决策、服务启动/退出等关键节点输出可诊断信息。
- SHOULD：日志输出应同时兼顾 **人类可读** 与 **机器可采集**（例如通过外部日志系统接入）。

## 2) 配置结构

- MUST：支持 `log` 配置段，结构示意：

```json
{
  "log": {
    "level": "info",
    "timestamp": true,
    "output": "console",
    "disabled": false
  }
}
```

> 注：字段命名/可选值以官方文档为准；实现时应严格对齐。

## 3) 功能需求

### 3.1 日志级别

- MUST：支持按级别过滤输出（至少覆盖：debug/info/warn/error 等常见分层）。
- SHOULD：在 `check` / 运行时分别提供“更偏静态语义/更偏运行链路”的日志提示。

### 3.2 时间戳

- MUST：支持 `timestamp` 开关；开启后每条日志包含可解析时间戳。
- SHOULD：时间戳格式应稳定（便于 log pipeline 解析）。

### 3.3 输出目标

- MUST：支持 `output` 指向控制台或文件（或等价抽象）。
- SHOULD：输出到文件时，至少保证：  
  - 目录不存在时的明确报错  
  - 权限不足时的明确报错  
  - 长期运行不导致无界增长（可交给外部 logrotate，也可内置策略，视实现选择）

### 3.4 全局禁用

- MUST：支持 `disabled`，用于在特定场景（例如嵌入式/极简环境）关闭日志输出。

## 4) 验收清单

- `disabled=true` 时应静默（但不影响程序退出码与错误返回）。
- 不同 level 下输出内容与数量符合预期。
- 配置错误能指向具体字段路径（例如 `log.level`）。
