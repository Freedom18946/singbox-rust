## 1. 事实：/connections 数据从哪里来

### 1.1 /connections HTTP/WS 的实现

`crates/sb-api/src/clash/websocket.rs`:
- `build_connections_snapshot()` 直接读取 `sb_common::conntrack::global_tracker().list()`
- 每条连接 JSON 取:
  - `"chains": c.chains`
  - `"rule": c.rule.unwrap_or("")`
  - `"metadata.type": c.inbound_type.unwrap_or("")`

`crates/sb-api/src/clash/handlers.rs`:
- `GET /connections` (无 WS Upgrade) 返回 `build_connections_snapshot()`
- `DELETE /connections/:id` 调用 `tracker.close(ConnId)`
- `DELETE /connections` 调用 `tracker.close_all()`

结论:
- `/connections` 是否有数据，完全取决于运行时是否有人往 `sb_common::conntrack` 注册连接。
- `/connections` 的关闭能力是否有效，完全取决于注册方是否把 `cancel` 传播到真实 I/O 循环。

### 1.2 ConnMetadata 的字段定义

`crates/sb-common/src/conntrack.rs`:
- 关键字段已存在:
  - `rule: Option<String>`
  - `chains: Vec<String>` (注释: "reversed: leaf → ... → matched outbound")
  - `inbound_type: Option<String>`
  - `cancel: CancellationToken`

## 2. 事实：当前谁在写入 global_tracker

工作区内对 `ConnMetadata::new(...)` 的唯一使用点是:
- `crates/sb-core/src/router/conn.rs`:
  - `ConnectionManager::new_connection()` (TCP) 在 dial 成功后注册 tracker
  - `ConnectionManager::new_packet_connection()` (UDP) 在 socket 创建后注册 tracker

但仓库中没有任何对以下方法的调用点:
- `ConnectionManager::new_connection(...)`
- `ConnectionManager::new_packet_connection(...)`

也就是说:
- 目前 `/connections` 极大概率为空。
- 目前 `DELETE /connections*` 即使返回 204，也不会真正关闭“实际跑着的连接”（因为没有注册或没传播 cancel）。

这点与 `agents-only/05-analysis/L2.8-CONNECTION-TRACKER-GAP-ANALYSIS.md` 中的 Go 数据流相比，缺少了最关键的 wiring:
- Go: Router.routeConnection() 后会遍历 trackers，把连接包一层计数 wrapper，并把 rule/chain 写入 TrackerMetadata。
- Rust: 目前只有 tracker 数据结构与 API 端点，缺少把真实连接接进去的路径。

## 3. 事实：哪些代码路径在建立真实连接

`crates/sb-adapters/src/inbound/*` 普遍采用如下模式:
- 使用 `rules_global::global().decide(RouteCtx)` 得到 `Decision`
- 根据 Decision 选择:
  - `outbounds.connect_preferred/connect_io(...)` 或
  - 环境变量 default proxy / pool selector 等自定义连接方式
- 用 `sb_core::net::metered::copy_bidirectional_streaming_ctl(...)` 做双向拷贝

关键点:
- `copy_bidirectional_streaming_ctl(...)` 已支持 `cancel: Option<CancellationToken>` (返回 Interrupted)。
- 但各 inbound 目前调用时 `cancel=None`，也没有任何 `global_tracker.register/unregister`。

结论:
- L3.5 “填充 rule/chains”如果只改 `sb-core/router/conn.rs`，很可能对真实 inbound 流量无效。
- 想让 `/connections` 真正可用，必须在真实 I/O copy 点做 tracker wiring，或把 inbounds 迁移去统一使用 `sb-core/router/conn::ConnectionManager`。

## 4. Go 对齐口径（用于 L3.5 的验收参考）

来自 `agents-only/05-analysis/L2.8-CONNECTION-TRACKER-GAP-ANALYSIS.md`:

- Chain 构建逻辑 (Go):
  - 从 matched outbound tag 开始
  - 若 outbound 是 group，则 `group.Now()` 得到 next tag，继续追到 leaf
  - 最后 `Reverse(chain)` => `leaf -> ... -> matched`
- `rule` 字段:
  - Go 输出 `"ruleString => action"` 或 `"final"`

## 5. L3.5 的阻塞点与需要先定的选择

### 5.1 阻塞点 A: tracker 没有接到真实连接

要交付“connections 面板可见 rule/chains”，先要保证 connections 面板确实有 connections:
- 选项 A: 推动 inbounds 统一走 `sb-core/router/conn::ConnectionManager`
  - 优点: 集中统一; cancel/计数/注册都在一处
  - 风险: inbounds 当前的连接策略分散（outbounds registry + pool + env default proxy），迁移可能扩大改动面
- 选项 B: 在 `sb-adapters` 的每个 inbound 的 copy 点做轻量接线（推荐作为 MVP）
  - 做法: 建连成功后注册 tracker，把 cancel 传给 `copy_bidirectional_streaming_ctl`
  - 优点: 改动局部、可渐进; 先把 `/connections` 变成“真的可用”
  - 风险: 需要统一抽一个 helper，避免每个 inbound 手写一套易错的 register/unregister/close 逻辑

### 5.2 阻塞点 B: rule 的“命中规则”目前拿不到

当前路由决策 API 只返回 `Decision`，没有 matched-rule 信息:
- `sb-core/router/engine.rs::RouterHandle::decide(...) -> Decision`:
  - 对 `idx.rules: Vec<CompositeRule>` 线性扫描，命中即返回 decision，但不返回 index/描述
- `sb-adapters` 多数走 `rules_global::global().decide(RouteCtx) -> Decision`，同样无 matched-rule 元信息

因此要产出 `ConnMetadata.rule`，需要新增一种“带元信息”的决策入口，例如:
- `decide_with_meta(...) -> (Decision, RuleMeta)`
- 或扩展 `RouteResult`/`DecisionMeta` 把 `matched_rule` 传播到调用者

### 5.3 阻塞点 C: chains 的“group -> member”需要能读取 group.Now()

链路对齐 Go 需要:
- 从 matched outbound tag 开始，按 tag 查询 outbound 实例
- 若 outbound 是 group（Selector/URLTest/LoadBalance/Fallback），调用 `OutboundGroup::now()` 得到下一跳 tag

Rust 侧具备的基础设施:
- `crates/sb-core/src/adapter/mod.rs` 定义了 `OutboundGroup` trait 与 `as_group()`
- `crates/sb-core/src/outbound/selector_group.rs` 已实现 `OutboundGroup`，`now()` 可用
- `crates/sb-core/src/outbound/mod.rs` 的 `OutboundRegistryHandle` 能按 tag 查 `OutboundImpl`
  - 若是 `OutboundImpl::Connector(conn)`, 则可 `conn.as_group()`

缺口:
- 需要一个通用 helper: `compute_proxy_chain(outbounds, matched_tag) -> Vec<String>`，带循环保护与 reverse。

## 6. 建议的 MVP 交付切分（把风险压低）

### 6.1 MVP-1: 先让 /connections “有真实连接 + 可关闭”

推荐先做选项 B（在 sb-adapters inbound copy 点接线）:
- 建连成功后:
  - `tracker_id = global_tracker.next_id()`
  - 构造 `ConnMetadata`:
    - `inbound_type`: "http"/"socks5"/"mixed"/...
    - `inbound_tag`: cfg.tag
    - `outbound_tag`: 实际使用的 outbound tag (或 proxy kind)
    - `chains`: 至少包含一个元素（例如 outbound_tag），后续再做 group->member 追踪
    - `rule`: 可以先留空（验收口径允许 rule/chains 至少一个非空）
  - `handle = tracker.register(meta)`
  - `cancel = handle.cancel.clone()`
- copy:
  - 调用 `copy_bidirectional_streaming_ctl(..., cancel=Some(cancel), ...)`
- finally:
  - `tracker.unregister(tracker_id)`

这一步完成后:
- `/connections` 不再是空壳
- `DELETE /connections/:id` / `DELETE /connections` 可以实际中断 I/O（Interrupted）

### 6.2 MVP-2: chains 对齐 Go（group->member->...->leaf, reverse）

在 MVP-1 基础上:
- 若使用 `OutboundRegistryHandle` 且决策结果是 `Proxy(Some(tag))`:
  - `chains = compute_proxy_chain(outbounds, tag)`
- fallback 到非 registry 的 default proxy/pool:
  - `chains = vec![actual_proxy_kind_or_tag]`

### 6.3 MVP-3: rule 对齐（至少可定位）

建议先实现“可定位但不重”的 rule 字符串:
- 对 composite rules: `"rule#<idx> => <decision>"`
- 对 default: `"final"`
- 后续再增强为更可读的 `"domain_suffix(x) => route(y)"`（可选）

## 7. 结论（本次 prework 的输出）

L3.5 真正的前置问题不是“字段怎么填”，而是:
1) `/connections` 的数据源 `global_tracker` 目前没有接入真实 inbound 流量
2) 即使接入，rule/chains 的元信息也需要在“路由决策”和“outbound group 选择”阶段被产出并传播

建议按 MVP-1 -> MVP-2 -> MVP-3 递进，先让面板变“可用”，再逐步逼近 Go parity。
