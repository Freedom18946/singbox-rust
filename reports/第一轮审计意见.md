下面是**第一轮审计意见（基于当前资料 + 代码抽查）**。我会把话说得很“杠精”，因为现在最危险的问题是：**项目文档里存在“自我证明闭环”，但代码里仍有关键能力处于 scaffold/stub 状态，却被一些文档宣称为“完成/生产可用”。**

------

## 0. 总体结论（先把最要命的点讲清楚）

### Critical 总览（不解决就很难达成“替代 sing-box”最高目标）

1. **“功能对等 / 生产可用”的公开叙述与代码现实不一致**：至少在 **TUN（macOS tun2socks）/ redirect/tproxy wiring** 等关键能力上，存在明显“文档宣称 ✅，实现却是 stub/scaffold/未接线”的断裂。
2. **uTLS 指纹 / ECH 的“对齐宣称”明显高估**：你们当前的 uTLS 更像“cipher suite/ALPN 微调”，与 Go `uTLS` 的能力边界差距很大；ECH 则还受制于 rustls/crypto provider/QUIC 组合问题，且 server-side ECH 现实上并不稳。
   - Go uTLS 的核心卖点就是可模拟/复刻多种 ClientHello 指纹（含扩展顺序等），这是 rustls 默认设计哲学不提供的能力。([Go Packages](https://pkg.go.dev/github.com/refraction-networking/utls))
   - rustls 的 ECH 目前明确是“实验性 client-side”，server-side 仍是单独议题/追踪中。([Prossimo](https://www.memorysafety.org/blog/rustls-ech-support/))
3. **架构宪法（Dependency Constitution）与当前 crate 现实严重背离**：`sb-core` 同时承担引擎层+协议层+服务层+控制面部分职责，且含大量 Web/TLS/QUIC 依赖；这会直接拖垮可维护性与边界治理，也让“Ports/clean architecture”变成口号。
4. **测试与验收口径偏控制面/GUI 兼容，缺少真实 TUN 数据面压力验证**：Interop Lab 的模拟很有价值，但它无法替代真实 TUN 场景（MTU、分片、重排、长时稳定性、路由回环、系统 DNS/进程匹配联动）带来的问题暴露。

------

## 1) Parity vs Reality：对齐目标与 Rust 生态现实冲突

### [Critical] 1.1 TUN（尤其 macOS tun2socks）在代码层面是“可编译 stub”，却被部分文档描述成“完成/完全支持”

【问题描述】

- 根 `Cargo.toml` 用 `[patch.crates-io]` 将 `tun2socks` 替换为 `vendor/tun2socks`，而该 vendor 实现是**明确的 build-time stub**（`main_from_str` 直接 `Ok(())`，`quit()` no-op）。
- `sb-adapters` 的 `tun_macos` 代码路径调用 `tun2socks::main_from_str(...)` 启动线程，但在 stub 下**不会产生任何实际转发能力**。
- `docs/07-reference/platform-io.md` 倒是诚实地说了 “scaffold TUN inbound remains a stub”，但 `agents-only/01-spec/01-REQUIREMENTS-ANALYSIS.md` 却将 `TUN, Redirect, TProxy` 标为 ✅，而 `README`/`docs/STATUS.md` 等又有“生产可用”的口吻，形成对外信息冲突。

【风险分析】

- 这是“替代 sing-box 核心用途”的硬伤：用户真正依赖的往往就是 TUN/透明代理链路。
- 更糟的是：**parity feature**（`app --features parity`）把 tun 相关 feature 也打进去了，但实际运行仍是 stub，属于“编译过关 ≠ 运行对等”。
- 文档层面“✅”会误导下游集成（GUI、发行版、packager）做出错误判断，最终变成“事故型开源项目”。

【具体建议】

- 明确做一个**三态能力矩阵**并强制落地（建议成为新的唯一事实来源）：
  - **Implemented & Verified（已实现且 e2e 验证）**
  - **Implemented but Unverified（实现了但没压测/没上真机）**
  - **Scaffold/Stub（只保证编译/接口，不保证运行）**
- `tun2socks` patch 建议改成**可切换**而不是全局覆盖：
  - `feature = "tun2socks-stub"` 默认给 CI；
  - `feature = "tun2socks-real"` 指向真实依赖（或 git tag）；
  - 并在构建产物/`--version` 输出中打印 capability（避免用户误用）。
- 如果你们目标是“Rust 完全重构 Go sing-box”，长期应避免依赖 Go tun2socks：
  - 要么把当前 `tun` scaffold 真正做成 Rust 数据面（smoltcp/自研 TCP/UDP session tracker），要么明确写入“de-scoped”。

------

### [Critical] 1.2 uTLS：当前实现并不等价于 Go uTLS 的“指纹模拟”，文档宣称存在明显高估

【问题描述】

- 代码层面（`sb-tls/src/utls.rs`）的“uTLS”主要做了：
  - cipher suite 列表/顺序的选择（以及大量“rustls 不支持就跳过”的过滤）
  - ALPN 列表设置
- 但 Go `uTLS` 的核心价值是更接近浏览器的 ClientHello 结构与细节（扩展集合/顺序/细粒度字段）。([Go Packages](https://pkg.go.dev/github.com/refraction-networking/utls))
- 对抗审查/指纹检测时，**JA3/JA3S 之类**看的是扩展、曲线、sigalg、顺序等综合指纹；仅动 cipher suite/ALPN 很可能远远不够。
- 你们的 README/用户文档中存在“27+ fingerprints”这类表述，与当前实现能力不匹配（至少需要“best-effort/partial”级别明确标注）。

【风险分析】

- 这会直接导致“用户以为能过指纹检测，但实际过不了”，属于高风险体验炸点。
- 更重要：为了追求“100% 对齐 uTLS”，很容易逼出 **异步架构扭曲**：
  - 在 rustls 现有 API 边界内强行 hack ClientHello（fork rustls 或 unsafe patch）会带来安全更新难度、维护噩梦。
  - 但又不能靠“纸面宣称”解决。

【具体建议】

- 先把目标定清楚：uTLS 在 Rust 侧可以有三条路，必须明确取舍：
  1. **承认 Partial**：把 uTLS 文档改为“cipher suite/ALPN 级别 best-effort”，并在 capability matrix 标为 ◐；
  2. **采用可定制 ClientHello 的 rustls fork/变体**：例如 `craftls` 这类项目定位就是“可自定义 ClientHello 指纹、预置浏览器指纹”。([GitHub](https://github.com/3andne/craftls))
     - 风险：安全更新跟进成本、审计成本；但这是最接近 uTLS 的路线。
  3. **改用 BoringSSL 生态（boring）**：走“成熟 TLS 库 + 可控握手”的路线。
     - 风险：跨平台构建复杂度（尤其和你们现在的 CI/离线目标冲突）。
- 不管选哪条路，都建议加一个**自动化指纹验收**：
  - 输出 ClientHello 的 JA3（或 equivalent）并对比 Go sing-box baseline（在相同配置下），把“对齐”从主观变成可测指标。

------

### [Critical] 1.3 ECH：文档/实现/依赖选择存在多重断裂（client-only、provider、QUIC 路径）

【问题描述】

- rustls 的 ECH 是“实验性 client-side”能力；server-side 仍在追踪/议题中。([Prossimo](https://www.memorysafety.org/blog/rustls-ech-support/))
- rustls 的 ECH 配置要求 HPKE provider 支持相应 suite，否则直接返回错误。([rustls.dev](https://rustls.dev/docs/client/struct.EchConfig.html))
- 你们在项目中同时存在：
  - 默认偏好 ring provider（多处 `install_default` 选择 ring）
  - ECH feature 下又引入 aws-lc-rs provider（因为 HPKE/ECH 需求）
- 这会触发一个非常典型的坑：**crypto provider 是进程级默认值，一旦安装默认 provider，后续很难切换**；而且当同时启用 ring + aws-lc 时，“谁优先”会非常关键（你们代码里目前倾向 ring）。

另外：QUIC 侧并非完全无路。Quinn 的 QUIC TLS client config 支持包装自定义 `rustls::ClientConfig`（前提 TLS 1.3），意味着理论上 ECH 可以通过自定义 rustls ClientConfig 带入 QUIC。([wide.gitlabpages.inria.fr](https://wide.gitlabpages.inria.fr/data-wallet-prototype/quinn_proto/crypto/rustls/struct.QuicClientConfig.html))
但你们当前 QUIC/ECH 路径在代码注释里是“简化版/未真正接入”，这属于“能力已在生态出现，但工程没对上”的典型逻辑缺口。

【风险分析】

- 用户配置了 ECH，结果因为 provider/feature 组合导致运行期 silently fallback / hard fail，会非常难排查。
- 更严重：你们文档里若暗示 server-side ECH 可用，会造成配置误导（现实上 rustls server-side ECH 不成熟/未完成）。([GitHub](https://github.com/rustls/rustls/issues/1980))

【具体建议】

- **把 crypto provider 选择上收为“单点决策”**：
  - 建议引入 `sb-crypto`（或在 `sb-tls`）作为唯一 provider 初始化入口；
  - 按 feature/运行配置决定 **ring vs aws-lc**，并且在启动时输出明确日志：“ECH enabled -> using aws-lc provider”。
- **QUIC + ECH 的实现路线要么做实，要么明确不支持**：
  - 如果要做实：走 “构造带 with_ech 的 rustls::ClientConfig → 转换为 QUIC client config” 的路线（Quinn 支持包装自定义 rustls 配置）。([wide.gitlabpages.inria.fr](https://wide.gitlabpages.inria.fr/data-wallet-prototype/quinn_proto/crypto/rustls/struct.QuicClientConfig.html))
  - 如果不支持：在 docs/STATUS/README 里明确写 “QUIC ECH not supported” 并给替代方案（例如只支持 TCP-TLS ECH）。
- 对外文档必须把 ECH 分成：
  - TCP/TLS ECH（client-side experimental）
  - QUIC ECH（是否支持、支持到哪一步）
  - server-side ECH（目前 rustls 现实）([GitHub](https://github.com/rustls/rustls/issues/1980))

## 2) Architectural Integrity：依赖宪法与 crate 边界审计

### [Critical] 2.1 “依赖宪法”写得很漂亮，但实现与检查脚本给了太多逃生舱（导致宪法形同虚设）

【问题描述】

- 架构规范（`agents-only/01-spec/03-ARCHITECTURE-SPEC.md`）明确写：
  - `sb-core` **禁止** web 框架（axum/tonic/tower）与 TLS/QUIC/WS 实现
- 但实际 `crates/sb-core/Cargo.toml` 里出现大量相关依赖（hyper/axum/tonic/quinn/rustls/reqwest…），并且 `sb-core/src` 里还直接包含 inbound/outbound/services/transport/tls 等目录，**已经是“巨石核心”**。
- `agents-only/06-scripts/check-boundaries.sh` 的检查策略是：
  - 只要依赖是 optional 或代码文件在白名单路径，就不报错
  - 这会让 “禁止依赖” 退化为 “只要 feature gate 就算合规”
- 结果：宪法与实际执行不一致 → 这是典型的架构治理逻辑断裂。

【风险分析】

- 长期维护会崩：
  - `sb-core` 的职责边界不清，任何功能都往 core 塞，最终你们会得到一个“比 Go 还难维护”的 Rust 项目。
  - feature gate 滥用导致组合爆炸，CI 很难覆盖所有组合，最终 runtime 崩溃只在用户机器出现。
- 你们会失去 Rust 的优势：本来应该通过 crate 边界做到“可替换、可测试、可裁剪”，结果做成一个“静态链接大一统”。

【具体建议】
这里必须做**二选一**（不选就是持续腐烂）：

**路线 A：真正贯彻宪法（推荐）**

- `sb-core` 只保留：router/rules/dns cache/session orchestration + Ports traits
- 把下面东西全部迁出 `sb-core`：
  - inbound/outbound 协议实现 → `sb-adapters`
  - transport/tls/quic/ws/http client → `sb-transport` / `sb-tls`
  - services/derp/v2ray/clash api → `sb-api` 或 `sb-services`
- 更新 boundary checker：禁止不仅是“非 optional”，而是 **禁止 default feature 间接启用**（要跑 `cargo tree -e features` 或基于 `cargo metadata` 做静态分析）

**路线 B：承认 sb-core 就是“内核合集”，重写宪法**

- 直接修改 spec：不再声称 `sb-core` 禁止 TLS/QUIC/Web
- 改成“模块化单仓”治理，而不是“分层架构治理”
- 但要相应更新命名（`sb-core` → `sb-engine` 或 `sb-kernel`），避免误导贡献者

------

### [Critical] 2.2 `sb-core` 与 `sb-adapters` / `sb-transport` 的功能重叠严重：重复实现会产生“分叉式腐烂”

【问题描述】

- `sb-core/src/inbound/*`、`sb-core/src/outbound/*`、`sb-core/src/transport/*` 与 `sb-adapters`、`sb-transport` 同名/同域模块并存。
- 这意味着：同一个协议/能力可能有两份实现（或半份实现），只靠 feature gate 决定走哪条路径。

【风险分析】

- 维护上会出现“修 bug 修到另一份实现没修”的典型事故。
- binary size/编译时间不可控；并且很难确认用户最终跑的是哪条实现路径（尤其当 GUI/packager 开启不同 feature 时）。

【具体建议】

- 做一次“协议实现归一化”决策：
  - **协议实现只允许存在一个 crate**（建议 `sb-adapters`）
  - `sb-core` 的 inbound/outbound 目录若要保留，只做 thin wrapper 或直接删除
- 在 CI 里增加 `--features parity` 的 “routing path snapshot” 输出：启动时打印每个 inbound/outbound 实际绑定的实现来源（core vs adapters），避免黑盒。

## 3) Operational Logic & Logs：运行逻辑、测试与验收口径

### [Risk] 3.1 互操作实验室（interop-lab）偏“控制面回放”，对 TUN/透明代理这类数据面风险覆盖不足

【问题描述】

- interop-lab 有大量 upstream fault 注入与 GUI replay，非常有价值；但它的测试模型主要在 TCP/HTTP/DNS/WS 等“可模拟”层面。
- 对 TUN 场景的核心痛点：
  - MTU、分片/重组、UDP/ICMP、路由回环、系统 DNS 争抢、长时间运行的 FD/内存泄漏、并发连接数压测
  - 很难通过纯模拟 upstream 覆盖

【风险分析】

- 项目容易陷入“GUI 能点、API 能回、但一开 TUN 就炸”的真实用户灾难。
- 这会直接背离你定义的“用户最高目标”（替换 sing-box 作为实际内核，而非仅 GUI 演示）。

【具体建议】

- 增加一个 **TUN e2e test profile**：
  - Linux：network namespace + veth + tun device + iptables 规则 → 做真实流量注入
  - macOS：至少提供一个“本机可重复”的长跑脚本（10k 连接 + UDP 流 + DNS 抖动）
- 给每个高风险能力（TUN/ECH/uTLS/REALITY/TLS-fragment）建立 **单独的 capability probe**，启动时输出明确状态（例如 “ECH: compiled=yes runtime=yes provider=aws-lc”）。

------

### [Critical] 3.2 验收标准中的性能/资源指标部分“过度乐观且定义不严”，会诱导错误优化方向

【问题描述】

- `agents-only/01-spec/02-ACCEPTANCE-CRITERIA.md` 中存在类似：
  - “原生进程匹配 <1ms（Go ~149ms）”
  - “连接开销 <1KB/conn”
  - “启动内存 <50MB”
- 这些指标缺少：测量方法（P50/P95/P99？冷启动还是热缓存？feature 开启集合？）以及平台范围（macOS-only？）。

【风险分析】

- 指标不严会导致：为了达标而做“不可维护的微优化”，最终损失稳定性。
- “<1KB/conn” 在 Tokio + TLS + QUIC + tracing + metrics 的现实里几乎不可泛化；达标往往意味着牺牲可观测性或功能。

【具体建议】

- 把验收指标拆成三层，并写死测量方法：
  1. **Baseline（默认路由/无 TUN/无 QUIC）**
  2. **Router+Clash API（GUI 实用配置）**
  3. **Parity（全特性）**
- 每层用同一套基准脚本输出：RSS、allocs、conn metadata size、P95 latency，并且将结果产物纳入 repo（防止口头“✅”）。

## 4) GUI & Ecosystem Coupling：与现有 GUI 的弱协议耦合风险

### [Risk] 4.1 GUI 兼容性靠“隐式契约”堆出来了，但缺少版本协商与契约测试的强约束

【问题描述】

- `agents-only/07-memory/LEARNED-PATTERNS.md` 里你们已经识别了很多 GUI 行为依赖点（比如对 `GLOBAL` 组、错误 JSON 形状、history key 等的严格期待）。这很好，但它本质上是**弱协议耦合**：GUI 更新一次就可能破。
- 如果未来出现通过“特定日志字符串”触发 GUI 状态变更的情况（很常见），风险会更高：日志本来是给人看的，不应成为机器协议。

【风险分析】

- GUI 更新会造成不可预期 break，最终用户只会说“这个内核不稳定”。
- 你们会被迫永久冻结行为细节，影响 Rust 侧演进。

【具体建议】

- 让“弱协议”变成“强契约”：
  - 增加一个 `GET /version` 或 `GET /capabilities`，返回：schema_version、clash_api_compat_version、feature flags、capability matrix
  - GUI 侧（或你们自己）写一个 contract test suite：固定 GUI 版本 + 固定请求集 → 校验响应 JSON shape（这比靠日志字符串稳）
- 日志：只做 human-readable；机器状态用结构化 endpoint / event stream（SSE/WS）。

## 5) 文档治理（额外但很关键）：当前存在多套“事实来源”，需要收敛

### [Risk] 5.1 文档之间相互打脸：Requirements/ROADMAP/STATUS/平台矩阵/代码注释并未统一口径

【问题描述】

- 你们同时维护：
  - `agents-only/*`（规划/验收/对齐矩阵）
  - `docs/STATUS.md`、`README.md`（对外宣称）
  - `docs/07-reference/platform-io.md`（相对诚实的能力矩阵）
- 其中对 TUN/redirect/tproxy/ECH/uTLS 的口径存在冲突：有的写 ✅，有的写 ◐/stub，有的写“production ready”。

【风险分析】

- 对外信誉风险：开源用户最怕“README 吹爆，跑起来残缺”。
- 对内工程风险：团队会用不同文档做决策，导致路线漂移。

【具体建议】

- 立一个“唯一事实来源”（Single Source of Truth）：`capabilities.json` / `capabilities.md`
  - 由 CI 自动生成（从 feature flags + probe + tests 结果汇总）
  - README/STATUS/agents-only 都引用它（不要手写多个版本）
- 把 “209/209 closed” 改成更诚实的口径：
  - “Config/API compatibility: X/Y”
  - “Runtime verified parity: A/B”
  - “Accepted limitations: …（含 uTLS/ECH/QUIC/TUN 等）”

------
